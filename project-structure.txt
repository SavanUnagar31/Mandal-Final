Mandal Backend Project Structure

The following is a complete directory structure of the Mandal backend project, including all folders and files, along with their purposes and usage in the context of a community finance management system. The project uses Node.js, Express, Sequelize, MySQL, Winston, Joi, JWT, bcrypt, nodemailer, Razorpay, Twilio, Redis, BullMQ, Swagger, Inversify, and Jest to handle user registration, mandal creation, member management, contributions, loans, and deployment with Docker and Kubernetes. Logs are stored in `logs/2025/August/29.log` format, and responses follow a standard format: `{ success: boolean, data: any, message: string }` for success and `{ success: false, error: string }` for errors.

# Root Directory: mandal-backend/
- Dockerfile
  Purpose: Defines the Docker image for the Node.js application.
  Usage: Used to build a containerized environment with Node.js 18, installing dependencies and running `server.js`. Enables consistent deployment across environments.
  
- docker-compose.yml
  Purpose: Configures Docker services for the application, MySQL, and Redis.
  Usage: Sets up local development environment with app, database, and cache services, specifying ports, environment variables, and volumes for persistence.

- .env
  Purpose: Stores environment-specific configuration variables.
  Usage: Contains sensitive data like database credentials, JWT secret, and API keys for SendGrid, Razorpay, Twilio, and Redis. Loaded by `dotenv` for secure configuration.

- .env.example
  Purpose: Template for `.env` file with placeholder values.
  Usage: Guides developers to set up required environment variables without exposing sensitive data.

- .gitignore
  Purpose: Specifies files and folders to exclude from Git version control.
  Usage: Ignores `node_modules`, `.env`, logs, and other temporary files to keep the repository clean.

- package.json
  Purpose: Defines project metadata, dependencies, and scripts.
  Usage: Lists dependencies (e.g., express, sequelize, winston) and scripts for starting the server (`npm start`), running tests (`npm test`), migrations, and seeding.

- server.js
  Purpose: Entry point of the application.
  Usage: Initializes Express app, sets up middleware, routes (`/api/v1`, `/api/v2`, `/api-docs`), database connection, and error handling. Starts the server on the specified port.

- README.md
  Purpose: Project documentation.
  Usage: Provides instructions for setting up, running, testing, and deploying the application, including Docker and Kubernetes commands.

# src/
Purpose: Contains the source code for the application, organized into subdirectories for API, domains, infrastructure, events, utils, config, and core.

## src/api/
Purpose: Defines API endpoints for v1 and v2 versions, including routes, controllers, middlewares, and validators.

### src/api/v1/
Purpose: Implements version 1 of the API with routes, controllers, middlewares, and validators for authentication, mandal, member, contribution, and loan functionalities.

#### src/api/v1/routes/
Purpose: Defines Express routes for API endpoints.

- index.js
  Purpose: Exports all v1 routes as a single module.
  Usage: Combines routes from `auth.routes.js`, `mandal.routes.js`, `member.routes.js`, `contribution.routes.js`, and `loan.routes.js` under `/api/v1`.

- auth.routes.js
  Purpose: Defines authentication-related routes.
  Usage: Handles POST requests for `/register`, `/login`, and `/verify` endpoints, using validators and controllers for user registration, login, and email verification.

- mandal.routes.js
  Purpose: Defines mandal-related routes.
  Usage: Handles POST (`/`), GET (`/:id`), PUT (`/:id`), and DELETE (`/:id`) for creating, retrieving, updating, and deleting mandals, with authentication and role-based middleware.

- member.routes.js
  Purpose: Defines member-related routes.
  Usage: Handles POST (`/:mandalId`), GET (`/:mandalId`), and DELETE (`/:mandalId/:userId`) for adding, listing, and removing mandal members, with admin role checks.

- contribution.routes.js
  Purpose: Defines contribution-related routes.
  Usage: Handles POST (`/:mandalId/pay`) and GET (`/:mandalId`) for paying and listing contributions, with member role checks and Razorpay integration.

- loan.routes.js
  Purpose: Defines loan-related routes.
  Usage: Handles POST (`/:mandalId/request`), PUT (`/:loanId/approve`), POST (`/:loanId/repay`), and GET (`/:mandalId`) for loan requests, approvals, repayments, and listing, with appropriate role checks.

#### src/api/v1/controllers/
Purpose: Contains controller logic for handling API requests.

- auth.controller.js
  Purpose: Handles authentication logic.
  Usage: Processes registration (creates user, sends verification code), login (issues JWT), and email verification, returning standardized responses.

- mandal.controller.js
  Purpose: Handles mandal-related logic.
  Usage: Manages creation, retrieval, update, and deletion of mandals, ensuring creator is set as admin and enforcing role-based access.

- member.controller.js
  Purpose: Handles member management logic.
  Usage: Adds, lists, and removes members from mandals, ensuring uniqueness and admin privileges.

- contribution.controller.js
  Purpose: Handles contribution-related logic.
  Usage: Processes payments via Razorpay and lists contributions for a user in a mandal, ensuring amount meets requirements.

- loan.controller.js
  Purpose: Handles loan-related logic.
  Usage: Manages loan requests, approvals (with interest calculation), repayments, and listing, ensuring proper mandal membership and status checks.

#### src/api/v1/middlewares/
Purpose: Contains middleware for authentication, authorization, validation, and error handling.

- auth.middleware.js
  Purpose: Verifies JWT tokens.
  Usage: Checks for valid JWT in the Authorization header, decodes it, and attaches user data to the request.

- role.middleware.js
  Purpose: Enforces role-based access control.
  Usage: Ensures the user has the required role (admin or member) for the mandal, checked via `MandalMember` repository.

- validation.middleware.js
  Purpose: Validates request bodies using Joi schemas.
  Usage: Applies validation schemas from `validators/` to ensure data integrity before passing to controllers.

- error.middleware.js
  Purpose: Handles errors globally.
  Usage: Logs errors with Winston and returns standardized error responses (`{ success: false, error: string }`).

#### src/api/v1/validators/
Purpose: Contains Joi validation schemas for API request bodies.

- auth.validator.js
  Purpose: Defines schemas for authentication endpoints.
  Usage: Validates `register` (name, email, mobile, password), `login` (email, password), and `verify` (email, code) requests.

- mandal.validator.js
  Purpose: Defines schemas for mandal endpoints.
  Usage: Validates `create` (name, contributionMode, contributionAmount, interestRate) and `update` (optional fields) requests.

- member.validator.js
  Purpose: Defines schema for member addition.
  Usage: Validates `add` request with `userId`.

- contribution.validator.js
  Purpose: Defines schema for contribution payment.
  Usage: Validates `pay` request with `amount`.

- loan.validator.js
  Purpose: Defines schemas for loan endpoints.
  Usage: Validates `request` and `repay` (amount) and `approve` (duration) requests.

## src/domains/
Purpose: Organizes business logic into domains (auth, mandal, finance) with entities, services, and repositories.

### src/domains/auth/
Purpose: Handles authentication-related logic.

#### src/domains/auth/entities/
- user.entity.js
  Purpose: Defines the User entity class.
  Usage: Represents a user with properties like id, name, email, mobile, password, and verified status.

#### src/domains/auth/services/
- auth.service.js
  Purpose: Implements authentication business logic.
  Usage: Manages user registration (checks uniqueness, hashes password, sends verification code), email verification (via Redis), and login (issues JWT).

- email.service.js
  Purpose: Handles email sending.
  Usage: Sends verification emails using nodemailer and SendGrid.

#### src/domains/auth/repositories/
- user.repository.js
  Purpose: Interacts with the User model in the database.
  Usage: Provides methods for creating, finding, and updating users, ensuring email/mobile uniqueness.

### src/domains/mandal/
Purpose: Manages mandal and member-related logic.

#### src/domains/mandal/entities/
- mandal.entity.js
  Purpose: Defines the Mandal entity class.
  Usage: Represents a mandal with properties like id, name, creatorUserId, contributionMode, contributionAmount, and interestRate.

- mandalMember.entity.js
  Purpose: Defines the MandalMember entity class.
  Usage: Represents a mandal membership with mandalId, userId, and role (admin/member).

#### src/domains/mandal/services/
- mandal.service.js
  Purpose: Implements mandal and member business logic.
  Usage: Handles mandal creation (sets creator as admin), retrieval, updates, deletion, and member management (add, list, remove with uniqueness and role checks).

#### src/domains/mandal/repositories/
- mandal.repository.js
  Purpose: Interacts with the Mandal model.
  Usage: Provides CRUD operations for mandals, ensuring name uniqueness.

- mandalMember.repository.js
  Purpose: Interacts with the MandalMember model.
  Usage: Manages member creation, role checking, listing, and deletion, ensuring unique memberships.

### src/domains/finance/
Purpose: Manages financial operations like contributions and loans.

#### src/domains/finance/entities/
- contribution.entity.js
  Purpose: Defines the Contribution entity class.
  Usage: Represents a contribution with properties like id, mandalId, userId, amountPaid, dueDate, paidDate, and status.

- loan.entity.js
  Purpose: Defines the Loan entity class.
  Usage: Represents a loan with properties like id, mandalId, userId, amount, interestRate, duration, and status.

- loanRepayment.entity.js
  Purpose: Defines the LoanRepayment entity class.
  Usage: Represents a loan repayment with properties like id, loanId, amount, date, and type.

#### src/domains/finance/services/
- contribution.service.js
  Purpose: Implements contribution business logic.
  Usage: Processes payments via Razorpay, lists contributions, and generates dues via cron jobs, ensuring amount validation and membership checks.

- loan.service.js
  Purpose: Implements loan business logic.
  Usage: Manages loan requests, approvals (with interest calculation), repayments, and listing, ensuring proper status transitions.

- payment.service.js
  Purpose: Handles payment processing.
  Usage: Creates Razorpay orders for contributions and loan repayments.

#### src/domains/finance/repositories/
- contribution.repository.js
  Purpose: Interacts with the Contribution model.
  Usage: Provides methods for creating and finding contributions by user and mandal.

- loan.repository.js
  Purpose: Interacts with the Loan model.
  Usage: Provides CRUD operations for loans, including listing by mandal.

- loanRepayment.repository.js
  Purpose: Interacts with the LoanRepayment model.
  Usage: Creates repayment records for loans.

# Summary
The project is structured to ensure modularity, scalability, and maintainability. The `src/api` directories handle RESTful endpoints with validators in `validators/` and route exports in `routes/index.js`. Business logic is encapsulated in `domains/`, with database interactions in `infrastructure/database/`. External services (email, payment, SMS) are abstracted in `infrastructure/external/`. Events and notifications are managed via `events/` and `infrastructure/queue/`. Utilities like logging (to `logs/2025/August/29.log`) and cron jobs are in `utils/`. Tests ensure reliability, and deployment scripts (`Dockerfile`, `docker-compose.yml`, `k8s/`) enable production-ready deployment.
